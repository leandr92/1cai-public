# Оценка архивных утилит: `cfg_tools` и `ones_universal_tools`

## Цель

Определить, какие возможности утилит @alkoleft стоит перенести в наш CLI, и подготовить дорожную карту миграции.

## Шаги исследования

1. **Сбор информации**
   - Изучить README/документацию репозиториев (`cfg_tools`, `ones_universal_tools`).
   - Зафиксировать поддерживаемые команды, формат конфигурации, требования по окружению (OneScript, EDT, внешние библиотеки).

2. **Функциональный аудит**
   - Сгруппировать функции по областям: сборка, выгрузка, анализ, миграция.
   - Определить востребованные сценарии в нашем стеке (CI/CD, parser pipeline, миграции).
   - Зафиксировать зависимости, которые сложно переносить (проприетарные компоненты, Windows-only сценарии).

3. **Прототипирование**
   - Выбрать 1–2 ключевые команды для PoC (например, выгрузка конфигурации, анализ зависимостей).
   - Подготовить минимальный Python-wrapper/CLI-вызов, интегрировать в Makefile.
   - Оценить время выполнения, совместимость с существующей инфраструктурой.

4. **План миграции**
   - Решить, переносим ли функциональность полностью или оборачиваем существующий инструмент (OneScript → Python subprocess).
   - Запланировать создание новых модулей в `scripts/cli` / `src/services`.
   - Определить требования к тестированию (unit + интеграционные).

5. **Документация**
   - Подготовить страницу с сравнением до/после.
   - Обновить `docs/01-getting-started` и `docs/architecture/README.md` после миграции.

## Критерии принятия решения

- Критичность функции для нашего pipeline (без неё не автоматизируем ключевой сценарий).
- Зависимости: возможно ли доставить без сложной установки.
- Поддерживаемость: есть ли активное развитие инструмента upstream.
- Лицензионные ограничения.

## Результат

После выполнения шагов подготовить отчет:
- Какие команды мигрируем в ближайший релиз.
- Какие оставляем как внешнюю зависимость (с обёрткой).
- Какие признаны устаревшими и не требуют переноса.
